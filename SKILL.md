# Database migrations (Alembic + NeonDB)

## Scope

This skill covers schema and data migrations for `investment-intelligence`:

- Authoring Alembic revisions (by hand or from autogenerate)
- Reviewing autogenerated diffs before they're trusted
- Running migrations against the correct NeonDB branch
- Backfills touching financial (NUMERIC) columns
- Rollout patterns for breaking schema changes

Not in scope: initial schema design, cross-database migrations, picking an ORM.

## Branch-aware execution (non-negotiable)

Production DB lives on `main`. Every feature branch maps to an isolated NeonDB branch. `DATABASE_URL` is resolved from the current git branch via the env loader — that's the guardrail that keeps you from clobbering production.

Before authoring or running any migration:

1. Confirm `git rev-parse --abbrev-ref HEAD` matches the expected Neon branch and that `DATABASE_URL` points at that branch DB.
2. If the two don't line up, **stop**. Flag it to the user. Do not override, stamp, or manually edit the URL.
3. Only after verification, proceed.

Rationale: Neon branches are cheap and disposable; `main` is not. The whole branch-aware workflow is pointless if migration commands bypass it.

## Authoring a migration

### Generate the revision

```bash
alembic revision --autogenerate -m "<imperative_snake_case_message>"
```

Good messages: `add_price_numeric_to_holdings`, `backfill_fx_rates_from_string`.
Bad messages: `migration`, `update_schema`, `fix`.

**Always review autogenerate output before committing.** Recurring failure modes in this project:

- **Missed `NUMERIC(precision, scale)` changes** — SQLAlchemy reflection treats `NUMERIC(18, 8)` and `NUMERIC(20, 10)` as interchangeable. Precision/scale drift is silent.
- **Drops ENUM types** it doesn't recognize (anything created via raw SQL in earlier migrations).
- **Ignores server defaults** set outside the model (e.g., `DEFAULT now()` added manually).
- **Reorders columns** on recreated tables in ways that break dependent indexes.
- **Adds redundant alter_column calls** when only comments or nullability changed.

If the diff doesn't match your mental model of what changed, trust your mental model and edit the file.

### Financial columns — precision is correctness

Every column storing a price, quantity, ratio, FX rate, or monetary value is `NUMERIC`, never `Float`:

```python
sa.Column("price", sa.Numeric(precision=18, scale=8), nullable=False)
```

`Float` on money is a data-integrity bug, not a style issue. If you see `Float` on a financial column in a migration you're editing or reviewing — including ones autogenerate proposes — flag it and fix it.

**Precision/scale: reuse existing conventions.** Don't invent new values per migration. Before adding a new financial column, grep for the closest existing analog:

```bash
grep -r "Numeric(" src/investment_intelligence/models/
```

Match the existing convention unless there's a specific reason not to — and if there is, explain it in the migration docstring.

### Data migrations & backfills

If a migration includes a data step:

1. **Split schema and data into separate revisions when feasible.** Easier to rollback the data step independently, easier to reason about.
2. **Coerce float→NUMERIC via string, not direct cast.** Direct numeric cast of `double precision` preserves the float representation error:
   ```python
   op.execute("""
       UPDATE holdings
       SET price_new = CAST(price_old::text AS NUMERIC(18, 8))
       WHERE price_new IS NULL
   """)
   ```
3. **Batch large UPDATEs.** A single UPDATE on millions of rows holds locks too long even on Neon. Use `WHERE id BETWEEN :lo AND :hi` and loop in the migration, or run the backfill out-of-band and use the migration only for the schema piece.
4. **Never `SELECT ... FROM` the ORM inside a migration.** Model classes drift; migrations must be self-contained. Use raw SQL or `sa.table(...)` definitions local to the migration.

### Indexes on large tables

Postgres `CREATE INDEX` takes an `ACCESS EXCLUSIVE` lock. For any table expected to grow beyond ~100k rows, use `CONCURRENTLY`:

```python
def upgrade() -> None:
    # CONCURRENTLY cannot run inside a transaction block.
    op.execute("COMMIT")
    op.execute(
        "CREATE INDEX CONCURRENTLY ix_holdings_symbol_ts "
        "ON holdings (symbol, ts DESC)"
    )
```

This requires either `transaction_per_migration = False` in `env.py` or the explicit `op.execute("COMMIT")` above. Confirm which your `env.py` uses before copy-pasting.

### Expand → contract for breaking changes

Any migration that renames or drops a column currently read by running code must be split across two deploys:

1. **Expand**: add the new column, dual-write from app code, backfill.
2. **Deploy** the app version that reads the new column.
3. **Contract**: drop the old column in a later migration.

Never do expand + contract in a single migration if an app is live against that DB. This is the single most common cause of mid-deploy outages.

## Downgrades

Every migration needs a working `downgrade()`. If reversal is genuinely impossible (dropped table, lost data, irreversible transformation), make it explicit:

```python
def downgrade() -> None:
    raise NotImplementedError(
        "Downgrade not supported: pre-migration price_old values are lost "
        "once the column is dropped. Restore from NeonDB point-in-time backup."
    )
```

A `pass`-body downgrade silently "succeeds" and lies to future-you. That's worse than raising.

Test the downgrade before merging:

```bash
alembic upgrade head
alembic downgrade -1
alembic upgrade head
```

If that round-trip fails, the migration isn't done.

## Review checklist (before opening the PR)

Before requesting review (or merging solo), walk through this list:

- Migration message is imperative and specific (not "migration" or "update_schema")
- Autogenerated SQL was reviewed against actual intent, not rubber-stamped
- No `Float` columns added for financial data
- `NUMERIC(precision, scale)` matches existing codebase convention
- `downgrade()` is implemented, or explicitly raises `NotImplementedError` with reason
- `upgrade → downgrade → upgrade` round-trip succeeded on the Neon branch
- Index creation on large tables uses `CONCURRENTLY`
- Breaking changes are split into expand/contract across separate revisions
- Data backfills on NUMERIC columns use string-cast, not direct cast
- No ORM model imports inside the migration file

## Execution

On the feature branch, after the Neon branch DB is seeded:

```bash
alembic upgrade head
alembic current         # verify revision applied
alembic history | head  # sanity-check recent history

# If something looks wrong
alembic downgrade -1
```

**Never `alembic stamp` to paper over drift** without surfacing it to the user first. Stamping marks a revision as applied without running it — it's a sharp tool that hides the actual problem 90% of the time you reach for it.

## Recovery

- **Drift between model and DB on a Neon branch**: run `alembic check`. If the output surprises you, read the last two or three revision files to reconstruct history before guessing.
- **Broken migration mid-run on a Neon branch**: don't surgically repair state. Delete the Neon branch, recreate from `main`, re-run. Neon branches are cheap — that's the whole point.
- **Broken migration on `main`**: stop. This is an incident. Don't run further migrations until the team (or, in solo mode, you after a break) has rolled back via Neon point-in-time restore and confirmed state.

## What belongs elsewhere, not here

Some migration discipline is better enforced mechanically:

- Blocking `sa.Float` in `migrations/versions/*.py` → **pre-commit hook**, not this skill.
- Blocking direct commits to `main` → **pre-commit hook**.
- Auto-resolving `DATABASE_URL` from git branch → **env loader**, not this skill.
- "Always use Conventional Commits" for the migration commit message → **commit-msg hook**.

This skill covers the judgment-heavy, context-specific parts. The mechanical parts should be in hooks so they can't be skipped.
